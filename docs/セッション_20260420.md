# セッションログ 2026-04-20 — DiskSage 誕生の日

## 発端

Mac（460GB SSD）の空き容量がまた 400MB まで減り、Zoom 中にメール受信ができなくなる症状が発生。Foresthill が Claude に相談したのが始まり。

## 症状

- 60GB 空けても数週間で空き容量ゼロに戻る
- Zoom 会議中にメールが受信できない二次被害
- パソコンを常にフル稼働、整理する時間と余裕がない
- バックアップ戦略ゼロ（iCloud/Google One も解約予定）
- NAS を持ちたいが所有していない

## 診断プロセス（再現性のあるフロー）

### Phase 1: ホーム配下の TOP 20

```bash
du -sh ~/* ~/.* 2>/dev/null | sort -hr | head -20
```

結果：Library 118GB（後 132GB まで膨張）、Documents 35GB、Development 33GB、Music 23GB、.ollama 11GB 等。

### Phase 2: Library ドリルダウン

```bash
du -h -d 1 ~/Library 2>/dev/null | sort -hr | head -15
```

> **macOS の du のクセ**: `-sh` と `-d N` は排他的で一緒に使えない。GNU du は通るが BSD du (macOS) は黙ってエラーで終わる。`-h -d 1` を使うのが正解。

結果：Application Support 78GB、Caches 24GB、Developer 14GB、Containers 7.1GB 等。

### Phase 3: Application Support の内訳

結果：iMobie 39GB、Claude 9.5GB、VoiceInk 8.1GB、Dia 3.9GB 等。

### Phase 4: iMobie の正体確認

iMobie 配下に `Backups/00008110-000459D92106201E/` というディレクトリ。UDID 形式。

```bash
plutil -p ~/Library/Application\ Support/iMobie/Backups/<UDID>/Info.plist | \
  grep -E "Device Name|Product Type|Phone Number|Last Backup"
```

結果：
- Device Name: "Vongole Bianco" ← Foresthill 自身の iPhone
- Product Type: iPhone14,6
- Last Backup: 2026-02-01

さらに `Status.plist` を確認：
```
"BackupState" => "empty"
"SnapshotState" => "uploading"
```

そして `Manifest.db` は存在しなかった。

> **重要な技術的発見**: iTunes/AnyTrans 系 iPhone バックアップは `Manifest.db` が復元用インデックス。これが無い = **バックアップ破損、復元不能**。

## 真犯人 vs 小犯人

Foresthill の直感通り、iMobie は**ストック型ゴミ**（2 月 1 日生成、以来放置）であって、「60GB 一瞬で埋まる」現象の犯人**ではなかった**。

### ストック型 vs フロー型の切り分け

- **ストック型**: 過去の残骸、一度削除すれば済む（iMobie 39GB、古い Ollama モデル、古いバックアップ）
- **フロー型**: 継続的に増殖する、根本対策が必要（VM swap、Docker.raw、日々のキャッシュ、業務ファイル）

DiskSage のキー機能の1つは、この2つを明確に区別して表示すること。

### フロー型犯人候補（本命）

Foresthill の Mac で見つかったフロー型：

1. **VM ボリューム 22.6GB**（macOS スワップ、再起動で解放）
2. **/Library 53GB**（ホーム外の共有アプリデータ、`~/Library` と別物）
3. **業務撮影データ（HHMS）月 8GB 級**（看板撮影の zip が溜まる）
4. **Claude Desktop VM 8.7GB**（Claude Code 実行環境、消してはいけない）
5. **Ollama モデル 10.7GB**（新モデル試すたびに増える）

### 「見えない 170GB」問題

- データボリューム使用: 400GB（`diskutil apfs list` より）
- ホーム配下: ~230GB（iMobie 削除後）
- 差分: 170GB がホーム外 → `/Library` 53GB、`/Applications` 33GB、`/private/var` 7.8GB 等
- **これは `df -h /` や `find ~` では一切見えない**

## DiskSage 設計への反映

この 1 日で判明したことを、すべてプロダクトの設計に反映した：

### 検出パターンのシード

| 今日の発見 | DiskSage パターン |
|---|---|
| iMobie 39GB 孤児化 | `iphone_backup`（Manifest.db 有無で破損判定） |
| VM swap 22GB | `vm_swap`（再起動推奨） |
| APFS スナップショット | `apfs_snapshots`（3個超で警告） |
| Ollama 10.7GB | `ollama_models` |
| Claude Desktop VM | `electron_cache`（VM は除外ルール） |
| node_modules 累積 | `node_modules_aggregate` |

### UI 設計への示唆

- 「シノギ型」と「フロー型」を別タブで表示
- 破損判定結果を必ず表示（「復元不能」の警告）
- Info.plist / manifest 読める場合はそれを表示
- **「誰の端末のバックアップか」を UDID から照合**（他人のデータは誤削除リスク）

### 信頼の根拠

Foresthill の「rm -rf 怖いな」という反応は、**商用ツール不信の市場ニーズそのもの**。DiskSage が「絶対に自動削除しない」を貫く根拠になった。
