# DiskSage 要件定義 v0.1

## 1. 機能要件（Functional Requirements）

### 1.1 スキャン機能（FR-01）

**目的**: ディスク上の容量肥大箇所を検出し、ユーザーに提示する

**詳細**:
- FR-01-1: macOS ホーム配下 `~/` を既定のスキャン対象とする
- FR-01-2: 既知のパターンライブラリに基づいて検出（Docker.raw, node_modules 等）
- FR-01-3: 各 finding に severity（high/medium/low/safe）を付与
- FR-01-4: 各 finding に推奨アクションを付与
- FR-01-5: レポートを Markdown 形式で生成
- FR-01-6: スキャン結果を JSON で永続化（後の AI 判定や UI で再利用）

### 1.2 提案機能（FR-02）

**目的**: 検出結果を人間が判断しやすい形で提示する

**詳細**:
- FR-02-1: 検出結果を severity 順に並べて表示
- FR-02-2: **削除操作は DiskSage 自身では実行しない**（CLI 版はコマンド提示のみ、GUI 版は明示承認が必要）
- FR-02-3: 各 finding に「なぜこの判定か」の根拠を明示

### 1.3 AI 判定機能（FR-03、Phase 2）

**目的**: Claude API を使って、ルールベースでは判定不能なファイルも評価する

**詳細**:
- FR-03-1: ファイル名・パス・メタデータ（サイズ、最終更新、拡張子）のみを API に送信
- FR-03-2: **ファイル内容は絶対に送信しない**
- FR-03-3: BYOK（ユーザー自身の API キー）を既定とする
- FR-03-4: API キーは Keychain（macOS）/ Credential Manager（Win）に保管
- FR-03-5: オフラインモード（ローカル LLM フォールバック）を Phase 3 で提供

### 1.4 トレンド機能（FR-04）

**目的**: 空き容量の時系列変化を記録し、「一瞬で埋まる」イベントを検出する

**詳細**:
- FR-04-1: `snapshot` コマンドで現在値を記録
- FR-04-2: cron による定期実行サポート（README で案内）
- FR-04-3: `trend` コマンドで時系列を表形式で表示
- FR-04-4: 前回との差分を表示、+1GB 以上の増加は警告色
- FR-04-5: Phase 3 でグラフ表示（GUI）

### 1.5 削除補助機能（FR-05）

**目的**: ユーザーが承認した削除を安全に実行する

**詳細**:
- FR-05-1: 削除は常にゴミ箱経由（macOS: `osascript` で Finder 経由、Linux: `gio trash`）
- FR-05-2: `rm -rf` を直接呼ぶ削除機能は**提供しない**
- FR-05-3: 完全削除（ゴミ箱を空にする）は DiskSage の責務外（OS の機能を使ってもらう）
- FR-05-4: 外付けドライブへの退避ウィザード（Phase 2、`rsync` ベース）

### 1.6 コマンドラインインターフェース（FR-06）

**詳細**:
- FR-06-1: `disksage scan` — スキャン実行
- FR-06-2: `disksage scan --ai` — AI 判定付きスキャン（Phase 2）
- FR-06-3: `disksage snapshot` — 現在値を記録
- FR-06-4: `disksage trend` — トレンド表示
- FR-06-5: `disksage help` — ヘルプ
- FR-06-6: `disksage version` — バージョン表示
- FR-06-7: 終了コード: 0 = 正常、1 = 引数エラー、2 = 実行エラー

## 2. 非機能要件（Non-Functional Requirements）

### 2.1 パフォーマンス（NFR-01）

- NFR-01-1: ホーム配下スキャンは 5 分以内に完了（500GB クラス SSD 想定）
- NFR-01-2: スキャン中はファイルシステムへの書き込み負荷を与えない（読み取りのみ）
- NFR-01-3: メモリ使用量は 500MB 以下

### 2.2 セキュリティ・プライバシー（NFR-02）

- NFR-02-1: オフラインモードでは**いかなるネットワーク通信も行わない**
- NFR-02-2: AI モードでもファイル内容は送信しない（ファイル名・パス・メタデータのみ）
- NFR-02-3: API キーは平文保存しない
- NFR-02-4: ログにユーザーの個人情報を含めない（パスは匿名化オプションを提供）

### 2.3 可用性・耐障害性（NFR-03）

- NFR-03-1: 個別ファイルの読み取り失敗でスキャン全体が停止しないこと
- NFR-03-2: Permission denied の頻発に耐える設計
- NFR-03-3: 部分的な結果でもレポート生成できること

### 2.4 保守性（NFR-04）

- NFR-04-1: パターン追加が OSS コントリビューターに可能な設計
- NFR-04-2: 判定ロジックはコードレビュー可能（ブラックボックス禁止）
- NFR-04-3: Phase 2 以降、パターンは JSON ファイルに外出し

### 2.5 互換性（NFR-05）

- NFR-05-1: macOS 12 以降対応（Ventura 以降で確実に動作）
- NFR-05-2: Apple Silicon / Intel 両対応
- NFR-05-3: Phase 3 で Windows 10/11、Ubuntu 22.04+ 対応
- NFR-05-4: 将来の macOS アップデートへの耐性（プライベート API を使わない）

### 2.6 アクセシビリティ（NFR-06）

- NFR-06-1: CLI 版はカラー出力の自動無効化（`NO_COLOR` 環境変数、パイプ時）
- NFR-06-2: GUI 版（Phase 3+）はスクリーンリーダー対応、ダークモード対応

### 2.7 ライセンス・法的要件（NFR-07）

- NFR-07-1: コアは Apache License 2.0
- NFR-07-2: 依存ライブラリのライセンスを SBOM で公開
- NFR-07-3: GPL ライセンスのコードは混ぜない（商用利用を阻害しないため）

## 3. ユーザーストーリー

### US-01: 空き容量が突然ゼロになった

> **開発者として**、夜中に空き容量が一瞬でゼロになる現象の原因を特定したい。
> **DiskSage の行動**: `disksage scan` で現状把握 + `disksage trend` で過去の急増タイミングを表示。「Docker.raw が 3 時間で +50GB」などの具体的な増加イベントを特定できる。

### US-02: 古いアプリのデータが残ってる

> **一般ユーザーとして**、半年前に使ったアプリのデータが残ってないか確認したい。
> **DiskSage の行動**: `~/Library/Application Support` 配下を走査し、最終アクセスが古くサイズが大きいものを検出。iPhone バックアップなど型が明確なものは破損判定も実行。

### US-03: node_modules を整理したい

> **開発者として**、200 個のプロジェクトのうち、もう使わない node_modules を削除したい。ただし現在開発中のものは触らないでほしい。
> **DiskSage の行動**: `~/Development` 配下の node_modules を列挙し、30 日以上ファイル更新のないプロジェクトを候補として提示。

### US-04: CleanMyMac が怖い

> **プライバシー志向のユーザーとして**、商用クリーナーに任せず、自分で判断して削除したい。
> **DiskSage の行動**: 判定の根拠を明示、削除操作は常にユーザー承認。OSS なので監査可能。

### US-05: Claude Code ヘビーユーザー

> **AI 開発者として**、Claude Desktop / Claude Code / Ollama / Whisper モデル等の巨大ファイルを管理したい。
> **DiskSage の行動**: AI ツールのキャッシュ・モデル・VM をそれぞれ別カテゴリで表示。特に「実行環境（消してはいけない）」と「キャッシュ（消してOK）」を明確に区別。

---

## 4. 制約条件

- C-01: macOS では Full Disk Access 権限が必要な領域あり、ユーザーに明示的に案内する
- C-02: サンドボックス環境（Mac App Store）には当面対応しない（柔軟なファイルアクセスを優先）
- C-03: root 権限が必要な領域（/System/Volumes など）は読み取り専用で扱う、変更しない

---

## 5. スコープ外（非対応）

- 積極的な自動削除
- 暗号化されたファイルの復号・中身解析
- クラウドストレージ（iCloud/Dropbox 等）の内部まで踏み込んだ最適化
- ウィルススキャン・セキュリティ対策機能
- バックアップ機能そのもの（BackupCoach として別プロダクトで検討）

---

## 6. テスト観点（概要）

- 単体テスト（Phase 2 で bats-core 導入）
- 実機テスト（Foresthill の Mac、および β テスターの環境）
- 誤検出率の測定（「これは消さない方がよかった」ケースの記録）
- パフォーマンステスト（大容量ディスクでのスキャン時間）
